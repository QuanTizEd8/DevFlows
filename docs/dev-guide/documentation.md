# Documentation

DevFlows documentation is built with Sphinx, MyST Markdown, the PyData Sphinx
theme, Sphinx Design, and Sphinx Copybutton.

## Documentation Layers

Guides : Hand-written documentation under `docs/guides`. Guides explain how to
use and maintain the project.

Generated reference : Ignored pages under `docs/reference`. These pages come
from workflow metadata and workflow YAML and are generated before Sphinx builds.

Examples : Checked-in example workflows under `tests/fixtures`. Generated
reference pages render these examples inline.

## Build Docs

```bash
task docs
```

This runs:

```bash
devflows docs
sphinx-build -b html docs docs/_build/html
```

For live preview:

```bash
task docs-serve
```

## Generated Reference Pages

Reference pages are generated by:

```bash
pixi run devflows docs
```

The generator reads:

- workflow metadata from `workflows/<workflow-id>/devflow.yaml`
- workflow interfaces from `workflows/<workflow-id>/workflow.yaml`
- examples from paths listed under `examples`
- scenarios from `tests.scenarios`

Do not commit or edit generated reference pages directly. Update the source
workflow, metadata, example, or generator instead.

## Writing Guide Pages

Keep guide pages practical:

- start with the common path
- link to generated reference pages for exact workflow interfaces
- explain why a convention exists when it affects maintainers
- include commands that can be run from the repository root
- avoid duplicating large generated tables

## Keeping Docs In Sync

`task lint` verifies reference documentation generation with:

```bash
devflows docs --check
```

That check generates reference pages into a temporary directory. It does not
compare against committed reference files, because `docs/reference/` is ignored.
If a Sphinx build needs reference pages, run:

```bash
pixi run devflows docs
```
